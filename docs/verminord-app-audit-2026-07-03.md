## 2. Is it linked to Notion? Does it sync on every Notion update?

**Linked: yes, but indirectly. Auto-sync: NO — the sync is dead.**

Verified facts from the database:

- `notion_sync_log` contains **14 rows total, all from June 6, 2026** (runs at 03:30 and ~08:45 UTC). Nothing since.
- `max(synced_at)` on every `notion_*` table = **2026-06-06 08:45–08:46 UTC** → the Notion mirror is **~4 weeks stale** as of this audit.
- There is **no scheduler anywhere**: no `pg_cron`/`pg_net` extensions installed, no Notion-related Supabase Edge Function (only `extract-log` and `sheets-sync` exist), no Vercel cron. The June 6 syncs were run manually (one-off scripts/sessions).
- The scene-5 header says *"Fra Notion-hub'en · oppdateres hvert 10. min"* — misleading. The **client** re-reads the Supabase mirror every 60s, but the mirror itself is never refreshed from Notion.

**"It syncs only the appointed data" — this part is true by design.** The pipeline only mirrors the chosen databases: Oppgaver & Prosjekter, Råvare & Partnere, Forhandlere & Salg, Korrespondanselogg (regulatory), Leverandører & Logistikk, Innholdsstrategi. Two of them (`notion_regulatory`, `notion_innholdsstrategi`) are empty because the source Notion databases were empty at the last sync.

## 3. Page 5 bug — root cause of the flashing and broken layout

Scene 5 ("Oppgaver, regulatorisk & partnere") is the Notion scene. From the production bundle:

1. **Scenes are unmounted and remounted on every rotation pass.** Each scene component is recreated when its index comes up.
2. The Notion scene runs **4 independent Supabase queries on every mount** (`notion_tasks`, `notion_regulatory`, `notion_partners`, `notion_forhandlere`), each starting from a `null` state.
3. While `rows === null`, each panel renders the placeholder *"Laster fra Notion-speil..."* — so **every ~64 seconds the scene appears half-empty, then content pops in section-by-section as the 4 queries resolve at different times.** That is the flash and the layout jumping.
4. Compounding it: the fade-out starts 550ms before the scene switch, then the new scene fades in from `translateY(8px)` — combined with the loading pop-in, scene 5 visually "double-flashes".
5. Nothing is cached between rotations, and there is no minimum-height/skeleton reservation, so the two-column grid reflows as data arrives.

This is purely a frontend data-lifecycle bug — the fix is to fetch once at app level, cache, and keep scenes mounted (details in the plan below).

## 4. Charts — how "real" is the data today?

Mixed, and this matches the request to move to fully real data:

- The dashboard boots from a **hardcoded demo dataset**: system list with fixed temp/pH/moisture values and "updated 03.06" timestamps, and chart series generated by a **seeded random-walk generator** (`bJ`/`yJ` in the bundle) for every time range (7D/21D/3M/6M/YTD/ALL).
- Every 5 minutes the app fetches real Supabase data (`logs`, `tasks`, `harvests`, `roadmap`) and **overlays** it: latest sensor values replace the fake ones, and a real chart series replaces the fake one **only where ≥2 real data points exist for that system/range**. Otherwise the synthetic curve keeps showing.
- The `logs` table is alive and healthy: mobile logging (`source='mobil'`) has entries **through today (2026-07-03)**, plus 218 digitized handwritten rows through June 6. So real chart data exists — the fake fallback just masks the gaps.
- **Google Sheets:** a `sheets-sync` Edge Function exists (push endpoint: Sheets → Supabase `logs`/`pre_compost_logs`), but **no rows with `source='google_sheets_sync'` have ever landed** — the Apps Script trigger on the Sheets side is either missing or never fired. The endpoint also has its API key **hardcoded in the function source**.

## 5. Security notes (worth fixing, not blocking)

- The PIN gate is client-side; all dashboard tables are readable with the public anon key regardless of the PIN. Fine for a kiosk, but treat the URL as public.
- `sheets-sync` API key hardcoded in the deployed function (should live in Edge Function secrets, and be rotated).
- Supabase advisors flag: permissive `USING (true)` write policies for authenticated users on several tables (`systems`, `deals`, `partners`, `events`, `batches`, `stock`, `pre_compost_logs`, `page_views`), public storage bucket `sop-documents` allows listing, `handle_new_user()` SECURITY DEFINER callable by anon, leaked-password protection disabled.

## 6. Upgrade plan — "finish it properly"

### Phase 1 — Bring the Notion sync back to life, automatically (highest priority)
1. Build a **`notion-sync` Supabase Edge Function**: pulls the 6 appointed Notion databases via the Notion API (integration token stored as an Edge Function secret), upserts into the `notion_*` tables, removes rows deleted in Notion, and writes a `notion_sync_log` row per run (including errors).
2. **Schedule it**: enable `pg_cron` + `pg_net` and call the function every 5–10 minutes. Optionally add Notion webhooks → Edge Function for near-instant updates, keeping the cron as a safety net. ("Every time Notion updates" in practice = webhook push + periodic reconciliation.)
3. **Surface sync health in the UI**: "Synkronisert for X min siden" badge driven by real `synced_at`, turning amber/red when stale (>30 min), so a dead pipeline is visible instead of silent.

### Phase 2 — Fix the page-5 flash and layout
1. Lift all Supabase reads into **one app-level data store** (fetch once per minute, or subscribe with Supabase Realtime on the `notion_*` tables). Scenes read from the store — no per-mount fetch, no `null` → content pop.
2. **Keep all 5 scenes mounted** and switch with opacity/transform only; content stays stable between rotations.
3. Show a skeleton only on true first load; afterwards always render the last known data. Reserve fixed panel heights so the two-column grid never reflows.
4. Correct the "oppdateres hvert 10. min" label to reflect the actual sync cadence from Phase 1.

### Phase 3 — 100% real charts (Supabase + Google Sheets)
1. **Delete the random-walk generator and the hardcoded demo systems.** Charts render only real series from `logs`/`pre_compost_logs`, bucketed per range; where data is missing, show an honest "ikke nok data ennå" empty state instead of a fake curve.
2. Drive the systems list and statuses from the `systems` table, not the hardcoded array with "03.06" timestamps.
3. **Google Sheets**: finish the existing pipe — add a time-driven Google Apps Script trigger that POSTs new rows to the `sheets-sync` endpoint (or fold a Sheets pull into the Phase-1 scheduled function). Move the API key into secrets and rotate it.

### Phase 4 — Design polish ("beautiful and clean")
1. Introduce design tokens (colors, spacing, type scale — keep IBM Plex Sans and the navy/teal identity) and one shared Card/Panel component so all 5 scenes look like one system.
2. Smooth crossfade between scenes (no fade-to-empty gap), a subtle per-scene progress indicator on the dots, consistent header, tabular numerals everywhere.
3. Proper empty/stale/error states for every panel; overflow clamping on long Notion titles.
4. QA pass at 1080p and 4K TV, plus a responsive fallback for phone/desktop (the canvas is currently a fixed 1920×1080 transform).

### Phase 5 — Security hardening (quick wins)
Rotate the sheets-sync key into secrets; tighten the permissive write policies; disable public listing on `sop-documents`; revoke anon EXECUTE on `handle_new_user`; enable leaked-password protection. Optionally swap the cosmetic PIN for Supabase Auth magic-link if real protection is ever needed.

### Practical note on implementation
The app's source code lives in the **private repo `Martin-starr/verminord-internal`**, which is not attached to this session (only `martin-starr/godmode` is). Phases 2 and 4 need that repo added to a session ("add Martin-starr/verminord-internal"). Phase 1, 3 (backend), and 5 can be done directly against Supabase (Edge Functions + cron + policies) without touching the frontend repo — and are the fastest way to make the dashboard truthful again.

**Suggested order:** Phase 1 → 2 → 3 → 4 → 5. Phase 1 alone fixes the biggest problem: the dashboard currently presents 4-week-old Notion data as "live".
